--- NETWORKING BASICS ---

Multicast
---- 
one to many communication where information is adress to a group of desitination computers simultaneously --- computers must subscribe to the broadcast address

Unicast
----
unicaste is a message that is sent from one device to another device --- peer to peer

IP
----
Internet Protocal --- the method or protcol by whiich data is sent from one computer to another on the internet or intranet -- each computer (host) has at least 1 IP address that uniquely identifies it --- can be multicast or unicast

UDP
---
User Datagram Protocal -- offers NO delivery verification when messages are sent --- UDP supports multicast


TCP
---
tcp enables 2 hosts to establish a connection and exchange streams of data --- TCP guarantees the delivery packets were recieved in the same order they were sent -- UNICAST


PIM
----
PIM - Protocol independent multicast --- routing protocol -- used to send from a single source to multiple destinations accross the network --- we use PIM with our multicast --- needs the enabled on the network


IGMP
-----
Internet Group Management Protocol -- used on a ipv4 -- used t to establish multicast group memberships -- integral part of IP multicast

multicast can sometimes be considered not secure --- can be "chatty" - takes up too much bandwidth on the network --- our equip doesnt use a lot of streaming so not too much bandwidth


VLAN
----
Virtual Local Area Network --- just a virtual LAN on a standard network -- limits the connectivity of other devices -- the connected devices are able to communicate as if they were attached to an isolated network


NTP
----
Network Time Protocol - the athena processors will reach out to the internet to sync the clock via NTP --- quantum does NOT use this


DNS / mDNS
----
domain name system --- mDNS is a localized, multicast version of DNS Lutron uses to discover athena processors.  a dedicated mDNS application can be used to browse mDNS services on the network. ---  mDNS can be used to discover processors without multicast --- similar to Bonjour on mac

wireshark ---> sniffing tool


SSH & SCP
----
secure shell protocols used by lutron app and lutron designer for DB transfer / support / files / etc --- we do a handshake between the lutron proc and the application -- if we cant get a secure connection, we won't allow the transfer to go through


--- HARDWARE ---


Ethernet Hub
---
broadcasts all traffic to all devices, non-programmable ---- not used on job sites anymore.   if information is sent in one port, it gets sent out all the ports of the hub --- all hosts connected would always get the information even if you wanted to send to one host

Layer 2 Ethernet Switch
---
smart forwarding, when managed can be programmable for VLANs --- used in applications previously done by ethernet hubs --- only send the traffic where it is addressed -- builds a table of who is connected to it, and the data only goes out to the specified device on the table
--- can be managed or unmanaged -- managed: can be administered and configured by IT --- can enable / disable multicast / ping, various protocols

Layer 3 Devices (Router / Switch)
---
able to route between VLANs and other networks, programmable and firewall capable --- a lot of them have network switches built in -- allows you to route between one network to antoher --- almost every athena job needs connection to the cloud - can utilize router to route to internet

Athena naming convention -- 
Athena is comprised of a System and a Place
- System (Database) - a group of products that control each other --- Athena is DATABASE PER SYSTEM** -- databases are smaller, limit exposure
- Place - a collection of athena systems under one customer group's control --- i.e. physical location

Quantum naming convention --
- Database / System -- equivalent to PLACE in athena ---- ONE DATABASE IS USED FOR THE ENTIRE PLACE in quantum -- Quantum is SINGLE DATABASE***
- Subsystem -- equivalent to SYSTEM in athena



IP Networks need several address to operate correctly
- 192.168.1.0 -- network address
- 192.161.1.255 -- broadcast address -- everything hears the message within the subnet
- ip address of the host / device  (192.168.1.XXX)
- default gateway (192.168.1.1) --- **usually the ip address of the router** --- access from one subnet/lan/network to another -- traffic leaves / passes from one network to another -- goes through the gateway
- subnet mask (255.255.255.0) --- defines how many devices / addresses can be in the subnet (lan/network)

network address has 2 halves
- network address, this is the first half that is the same across devices on the network (192.168.0.XXX)
- host address - this is the id of the device / host (.XXX)
- subnet determines the split between the network and the host


subnet -- 255 means the 2 octets of 2 addresses MUST MATCH in order for an ip address to be able to talk to one another -- 0 in the subnet means the octet can be 0-254 and the devices can talk to each other -----  255 in an ip address is reserved for broadcast --
if lower than 255 -- the ones above the number (e.g. 240) are the only ones that can talk to each other

if they ask you how many ip addresses we need --- make sure we get more than the 16 procs -- we need 1 for each device connected, laptop, ipad, athena touchscreen devices, etc.



DHCP
---
dynamic host configuration protocol --- ip address automatically assigned to devices when they connect to the network

DHCP or Static IP Settings
---
athena --> DHCP is recommended to keep the processors Online -- a DHCP reservation can be added to the sites DHCP server to keep the address the same -- this is not a lutron setting

if devices on the same lan have the same IP address -- one or more of the devices wont communicate --- if a device is not working, try taking the device (eg. processor) offline, and pinging the known IP of the device to see if something else responds
